# Testing Setup (pytest + TDD) — Requirements

**Дата:** 2026-06-13
**Статус:** черновик требований (вход для `ce-plan`)
**Тип:** инфраструктура разработки + рабочая практика

## Проблема и контекст

Тестов в проекте нет; `AGENTS.md` сейчас прямо запрещает «тесты сверх запроса» (раздел «Чего не делать»). Это осознанный пересмотр: код растёт, надвигается крупный сдвиг multi-user (`docs/brainstorms/2026-06-13-multi-user-digest-requirements.md`) — появляются SQLite, cost-трекинг, планировщик-тик, идемпотентность. Хочется (в порядке приоритета): **навести структуру**, **получить страховочную сетку для свободного рефакторинга** и **зайти в multi-user без страха сломать живых платных юзеров**.

Изначально вопрос звучал как «ДДД», но это была оговорка — речь про **TDD**, не Domain-Driven Design. Доменное моделирование (entities, aggregates, repositories, слои-границы) **сознательно вне scope**: оно противоречит философии `AGENTS.md` (функции не классы, stateless, без DI/фабрик). TDD, наоборот, отлично ложится на чистые функции проекта.

## Цели

- Поставить лёгкую тестовую инфраструктуру на `pytest`, не таща её в прод-зависимости.
- **Страховочная сетка сейчас:** характеризационные тесты на рисковое чистое ядро текущего кода — до начала рефакторинга под multi-user.
- **Дальше — TDD:** весь новый multi-user-код пишется test-first (red-green-refactor).
- CI прогоняет тесты, чтобы сетка реально ловила регрессии.

## Не-цели (сознательно вне scope)

- DDD / доменное моделирование, классы-сущности, слои domain/application/infrastructure.
- Тесты на I/O-фетчеры (wttr.in, CoinGecko, forex) и реальные HTTP-вызовы — кроме снэпшот-фикстур для парсера.
- Тесты, завязанные на `digest/content/news/topics.py` / `NEWS_GROUPS` как статический каталог — он удаляется в multi-user (темы уезжают в `USERS_CONFIG`).
- Строгий test-first на ретрофит старого кода; coverage-проценты как обязательный гейт — ceremony для проекта этого размера.

## Принятые решения

### Подход и порядок
**Сетка → потом TDD.** Сначала покрыть тестами рисковое существующее ядро как страховку, затем multi-user-код писать test-first. Прагматично: для старого кода test-after допустим, дисциплина red-green-refactor — для нового.

### Тулинг и структура
- `pytest`, вынесен в `requirements-dev.txt` (на Railway pytest не нужен — прод-`requirements.txt` не трогаем).
- Top-level каталог `tests/`, зеркалит `digest/` (не тесты рядом с файлами: чище деплой, де-факто стандарт Python).
- Импорт пакета `digest` из `tests/`: пустой `pyproject.toml` с `[tool.pytest.ini_options] pythonpath = ["."]` **или** корневой `conftest.py` — выбрать в плане.

```
tests/
  conftest.py
  fixtures/
    sonar_*.json
  content/
    news/
      test_parse.py
      test_prompt.py
    test_report.py
    test_telegram_html.py
```

### Что покрываем сеткой сейчас (приоритет — стабильные чистые поверхности)
- `digest/content/news/parse.py` — citations whitelist, парсинг ссылок, сборка блока. №1 по риску.
- `digest/content/telegram_html.py` + `digest/content/report.py` (`build_news_groups_html_list()` и сборка HTML по заданным входам).
- `digest/content/news/prompt.py` — подстановка темы в фиксированный скаффолд.

Заморозка `parse.py` и формата вывода зафиксирована в multi-user-доке → сетка на парсер остаётся валидной сквозь multi-user.

### Фикстуры парсера
Снять пару реальных ответов sonar через `scripts/openrouter_call.py`, очистить (без секретов/PII), закоммитить в `tests/fixtures/`. Основа для тестов `parse.py`.

**Снэпшоты — ассерты на свойства, не точное совпадение блока:** N ссылок ≤ 4, все URL из whitelist (`citations`/`search_results`/`annotations`), нет голого/неэкранированного HTML, формат блока соблюдён. Устойчивее к косметике, ловит реальные регрессии.

### TDD-цели на multi-user (писать test-first)
Чистая логика, контракт ясен заранее:
- «кому пора» — по локальному времени юзера + слот расписания.
- Идемпотентность доставки (`brief_deliveries`: не слать слот дважды).
- Парсинг/валидация `USERS_CONFIG` (JSON).
- `usage_cost()` (`digest/content/openrouter.py`) — извлечение cost.

### I/O-граница
Фетчеры и реальные HTTP-вызовы тестами не покрываем (контракт = «что вернёт API»). Максимум — мок `requests` точечно, если понадобится; по умолчанию полагаемся на снэпшот-фикстуры парсера.

### CI
Прогон `pytest` в GitHub Actions на push и PR — отдельный workflow, не трогая cron-`daily.yml`. На Railway тесты не гоняются.

### AGENTS.md
Убрать «тесты сверх запроса» из «Чего не делать»; добавить короткую секцию-конвенцию: `pytest`, `tests/` зеркалит `digest/`, dev-зависимости в `requirements-dev.txt`, тестируем чистое ядро (не I/O), снэпшоты — ассерты на свойства.

## Допущения (проверить в плане)

- `parse.py` и формат вывода остаются заморожены (из multi-user-доки) → сетка долговечна.
- Коммиты в main (фича-ветки по ситуации) → CI имеет смысл и на push в main, не только на PR.
- Реальные ответы sonar можно безопасно закоммитить как фикстуры после очистки.

## Затронутые области кода (ориентир для плана)

- Правки: `AGENTS.md` (раздел «Чего не делать» + новая секция про тесты).
- Источники для фикстур: `scripts/openrouter_call.py`.
